1. Visión General de la Arquitectura

BandSocial es una Single Page Application (SPA) construida en React. No tiene backend propio: utiliza Firebase como Backend-as-a-Service, Cloudinary para gestión de imágenes y EmailJS para el envío de correos.

diagrama┌──────────────────────────────────────────────┐ │ FRONTEND (React + Vite) │ │ Components │ Pages │ Contexts │ Hooks │ └──────────────────────┬───────────────────────┘ │ SDK / REST API ┌───────────────┼───────────────┐ ▼ ▼ ▼ ┌──────────┐ ┌───────────┐ ┌──────────┐ │ Firebase │ │Cloudinary │ │ EmailJS │ │Auth+DB+ │ │ (Media │ │ (Emails) │ │ Storage │ │ CDN) │ │ │ └──────────┘ └───────────┘ └──────────┘

Características Técnicas Clave

  • Lazy loading de todas las páginas con React.lazy() + Suspense
  • Code splitting automático por vendors (React, Firebase, UI)
  • Modo oscuro/claro por CSS variables + data-theme en el HTML root
  • Modo invitado vía localStorage + GuestContext
  • Realtime en chat y notificaciones vía onSnapshot de Firestore
  • PWA — manifest.json + Service Worker incluido

5. Flujos de Datos Principales

5.1 Flujo de Autenticación

flujo1. Usuario → /login → Login.jsx 2. signInWithEmailAndPassword(auth, email, pass) 3. Firebase Authentication valida → retorna UserCredential 4. auth.onAuthStateChanged listener → setUser(currentUser) 5. Si email no verificado → muestra EmailVerificationPrompt 6. Redirige → /publicaciones (feed principal) 7. Carga perfil → getDoc(db, 'perfiles', uid)

5.2 Flujo de Publicación con Imagen

flujo1. PublicacionesNuevo.jsx → usuario escribe + selecciona imagen 2. uploadToCloudinary(file) → POST a Cloudinary CDN 3. Cloudinary retorna { secure_url } 4. addDoc(collection(db, 'publicaciones'), { ...datos, imagenUrl }) 5. Firestore guarda el documento 6. notificationService.notify() → seguidores reciben notificación 7. onSnapshot listener → feed se actualiza en tiempo real

5.3 Chat en Tiempo Real

flujo1. Usuario A abre chat → ChatDock → ChatWindow 2. handleSend() → addDoc(collection(db,'chats',chatId,'messages'), msg) 3. Firestore almacena el mensaje 4. onSnapshot listener en ChatWindow → Usuario B ve el mensaje al instante 5. updateDoc(userChats) → actualiza lastMsg + lastAt 6. useUnreadChats hook → actualiza indicador de mensajes no leídos

7. Reglas de Seguridad

Las reglas de Firestore están en firestore.rules. Funciones auxiliares reutilizables:

javascript// Verifica que hay sesión activa function isAuthenticated() { return request.auth != null; } // Verifica que el usuario es el propietario del documento function isOwner(userId) { return isAuthenticated() && request.auth.uid == userId; } // Verifica que el usuario es administrador function isAdmin() { return isAuthenticated() && request.auth.token.email == 'estebanber24@gmail.com'; }

8. Gestión de Estado

Estado Local (useState)

Usado para UI de componente: modales, tabs, forms, loadings.

javascriptconst [showModal, setShowModal] = useState(false); const [loading, setLoading] = useState(true);

Estado Global (Context API)

ThemeContext persiste en localStorage, aplica data-theme al <html>:

javascriptdocument.documentElement.setAttribute('data-theme', theme); // 'light' | 'dark'

ChatDockContext controla las ventanas de chat abiertas:

javascriptconst { openChat, closeChat, openChats } = useChatDock();

GuestContext gestiona el modo visitante sin cuenta:

javascriptconst { isGuest, activateGuest, deactivateGuest } = useContext(GuestContext);

Estado del Servidor (onSnapshot)

javascriptuseEffect(() => { const unsub = onSnapshot( query(collection(db, 'notificaciones'), where('usuarioId', '==', uid), orderBy('createdAt', 'desc') ), (snap) => setNotificaciones(snap.docs.map(d => ({ id: d.id, ...d.data() }))) ); return unsub; // Limpia el listener al desmontar }, [uid]);
⚠️ Importante: Todos los onSnapshot deben retornar unsubscribe en el cleanup del useEffect para evitar memory leaks.

11. Guía de Contribución

Flujo de Trabajo Git

bash# 1. Crea una rama desde main git checkout -b feature/nombre-de-la-feature # 2. Commits semánticos git commit -m "Feature: Descripción corta" git commit -m "Fix: Descripción del bug corregido" git commit -m "Refactor: Descripción del refactor" # 3. Sube la rama y abre un Pull Request git push origin feature/nombre-de-la-feature

Convención de Commits

PrefijoUso
Feature:Nueva funcionalidad
Fix:Corrección de bug
Refactor:Refactorización sin cambio funcional
Docs:Cambios en documentación
Style:Cambios de estilos CSS
Test:Agregar o modificar tests
Revert:Revertir un commit anterior

Convenciones de Código

  • Componentes en PascalCase: ProfileCard.jsx
  • Hooks en camelCase con prefijo use: useNotifications.js
  • Servicios en camelCase: notificationService.js
  • Usar Vanilla CSS para estilos — no inline styles ni Tailwind
  • Exportar contextos con su Provider y hook use* en el mismo archivo
  • Limpiar siempre los listeners de Firestore (onSnapshot) en el useEffect cleanup
